# worktree-merge

当用户输入 `$worktree-merge 任务名` 或要求合并 worktree 任务时使用本 skill。

## 执行要求

- 必须在主项目目录执行。
- 从用户输入中提取任务名；任务名不能为空。
- 调用 `python3 "$HOME/.agents/skills/codex-cli-worktree/scripts/worktree-task.py" merge "任务名"`。
- 不要启动任何项目服务。
- merge 会在主项目目录干净或等于已记录的 switch 预览时，自动把主项目目录恢复到当前提交并清理新增文件，再把任务目录里的改动三方合并到主项目目录。
- 合并成功后提醒用户：在主项目目录手动重启服务验证，验证通过后由用户手动 commit。
- 如果脚本提示 Git 合并冲突，不要只复述文件列表，也不要让用户只能选择保留主项目版本或任务版本。
- 冲突发生后，Codex 必须继续进行 AI 语义分析：
  - 读取 `git diff --name-only --diff-filter=U --` 获取冲突文件列表。
  - 读取 `git status --short --untracked-files=all` 和 `git diff --name-status HEAD --` 获取本次合并已经写入工作区的完整文件范围，不能只看冲突文件。
  - 对每个冲突文件读取工作区冲突内容，必要时读取 `git show :1:路径`、`git show :2:路径`、`git show :3:路径` 分别查看 base、主项目版本、任务版本。
  - 围绕冲突文件继续检索相关代码：路由、菜单、store、API、权限、配置项、数据库迁移、测试、文档、导入引用和调用方。需要用 `rg` 或现有项目工具查找相关符号、路径和配置键。
  - 必须做整体一致性分析，说明每个方案对前端入口、后端 API、权限/授权、配置状态、数据库 schema、文档和测试的影响；避免只解决冲突标记但留下断路由、空引用、重复入口、缺失权限、配置读写不一致或迁移顺序问题。
  - 分析双方意图、冲突原因、跨文件风险点，并为每个文件或文件组给出 2-3 个语义级修复方案，例如“两边入口都保留并调整顺序”“系统配置页只保留主题，迁移配置到独立页”“三条 changelog 都保留并按时间排序”。
  - 推荐方案必须包含整体检查项：哪些相关文件需要一起改、哪些命令或测试建议用户之后验证、还有哪些风险需要用户确认。
  - 提供一个整体选项“全部手工处理”，以及逐文件选择入口。
  - 用户选择某个方案后，Codex 使用编辑工具直接修改对应冲突文件，删除冲突标记，并在适当时执行 `git add` 标记该文件已解决。
- 若用户选择全部手工处理或某个文件手工处理，保留对应冲突状态并提醒用户后续手动解决、验证后手动 commit。
- 如果脚本提示主项目目录存在非 switch 产生的未提交改动，停止并提示用户先提交、stash、清理或手动处理。
- 遇到数据库 schema、路由、权限、授权、配置、状态机等语义冲突，必须询问用户，不要擅自决定。
